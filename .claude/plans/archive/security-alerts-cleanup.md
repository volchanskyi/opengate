## Approach

Two independent fixes in two commits — different scopes, different verifications.

### Commit 1 — Tighten test-fixture file permissions to satisfy `go:S2612`

CLAUDE.md SonarCloud workflow forbids new `sonar.issue.ignore.multicriteria` suppressions without explicit approval. The principled fix is to narrow the perms; precedent already exists at [cert_test.go:121](server/internal/cert/cert_test.go#L121) (`os.Chmod(..., 0600)`) which Sonar does not flag. The tests use `t.TempDir()` (process-isolated, auto-cleaned), so owner-only perms are safe and preserve test semantics.

Changes to [server/internal/cert/cert_test.go](server/internal/cert/cert_test.go):

| Line | Before | After  | Why |
|------|--------|--------|-----|
| 110  | `0644` | `0600` | Cleanup chmod restoring caCertFile perms; owner-read+write is enough for `t.TempDir()` to clean up. Matches sibling line 121. |
| 129  | `0555` | `0500` | Test body — making dir non-writable. `0500` (owner r-x, no group/other) still satisfies the "fails to write to read-only dir" assertion because no one can write. |
| 130  | `0755` | `0700` | Cleanup restoring dir perms. Owner r-w-x is the minimum for `t.TempDir()` to recurse and remove the dir. |

No suppression added to [sonar-project.properties](sonar-project.properties).

### Commit 2 — Bump transitive `rand` to 0.9.3 to patch RUSTSEC-2026-0097

`rand` is not declared in any `Cargo.toml` — it is pulled transitively. A cargo update with `--precise` updates only the lockfile, no source changes needed. Same pattern as recent commit `aaae2dc fix(deps): bump rustls-webpki to 0.103.12 to patch RUSTSEC-2026-0098/0099`.

```bash
cd agent && cargo update -p rand --precise 0.9.3
```

Updated file: [agent/Cargo.lock](agent/Cargo.lock) only.

## Notes / decisions

- **Why fix vs suppress for `go:S2612`**: CLAUDE.md SonarCloud section explicitly says "Do not add `sonar.issue.ignore.multicriteria` entries... unless the user explicitly approves." Tightening perms is the principled fix and matches the unflagged pattern at line 121.
- **Why two commits, not one**: The two changes have nothing in common (Go test perms vs Rust lockfile bump) and different verification paths. Splitting keeps `git blame` and revert granularity clean.
- **Per CLAUDE.md**: skip `/precommit` is at the user's discretion. Commit 2 is a pure lockfile bump; Commit 1 is a 3-line test-file change. Both are low-risk but not zero — the user should decide whether to run the full battery (will require Postgres up + SONAR_TOKEN, ~10 min).
