## 2. CI Workflow Permissions (3 alerts)

### 2a. Move permissions from workflow level to job level
**File:** `.github/workflows/ci.yml`
- Alerts 42-44: `githubactions:S8233` (write) and `S8264` (read) — permissions should be scoped per-job
- Move the workflow-level `permissions:` block (lines 12-16) down to each job that needs them
- Jobs needing `security-events: write`: the SonarCloud/CodeQL job
- Jobs needing `contents: read`: most jobs (checkout)
- Jobs needing `issues: write`: dependabot auto-merge or similar
- Jobs needing `actions: read`: any job reading workflow artifacts

**Approach:** Remove workflow-level `permissions:` block. Add minimal `permissions:` to each job. Read full ci.yml to determine exact per-job needs before implementing.

**Resolves:** 3 alerts

### 3b. `godre:S8242` — Remove context.Context struct fields (2 alerts)
These are legitimate — Go best practice says context should be passed as a parameter, not stored in structs.

**File:** [wsconn.go](server/internal/api/wsconn.go) lines 14-15
- Remove `ctx` and `cancel` fields from `WSConn` struct
- Change `NewWSConn(conn)` signature to `NewWSConn(ctx context.Context, conn)`
- Store only the `cancel` func (needed for `Close()`), pass `ctx` via method params
- Update `Read()` and `Write()` to accept `ctx` parameter — BUT `WSConn` implements `io.ReadWriteCloser` which has fixed signatures
- **Alternative:** Since `WSConn` must satisfy `io.ReadWriteCloser` (no ctx param on Read/Write), the stored context is architecturally necessary. Suppress this specific alert with `//nolint` or add a SonarCloud exclusion for this file+rule. Document why context storage is required (interface constraint).

**File:** [relay.go](server/internal/relay/relay.go) lines 43-44
- `session` struct stores `done <-chan struct{}` and `cancel context.CancelFunc`
- The `done` channel is derived from a context but is `<-chan struct{}` not `context.Context` — this may be a false positive since the field type is `<-chan struct{}`, not `context.Context`
- The `cancel` field is `context.CancelFunc` which SonarCloud may flag as context-related
- **Approach:** Refactor to pass context through `pipe()` call chain. Create context in `Register()`, pass to `go r.pipe(ctx, token, s)` instead of storing on struct. Keep `cancel` on struct since it's needed in `defer` cleanup.

### 3c. `go:S3776` — Reduce cognitive complexity (1 alert)
**File:** [handlers_relay.go](server/internal/api/handlers_relay.go) `handleRelayWebSocket` (lines 26-96)

```go
func (s *Server) extractBrowserAuth(r *http.Request) string {
    header := r.Header.Get("Authorization")
    if header == "" {
        if authParam := r.URL.Query().Get("auth"); authParam != "" {
            header = "Bearer " + authParam
            r.Header.Set("Authorization", header)
        }
    }
    return header
}
```

### 3d. `godre:S8193` — Unnecessary variable declarations (14 alerts)
These alerts reference line numbers that may not match current code (code has been refactored since scan). The pattern SonarCloud flags is:
```go
// Flagged: err declared but not used in body
if err := s.store.Ping(ctx); err != nil {
    return ..., nil  // err not referenced here
}
// Fix:
if s.store.Ping(ctx) != nil {
    return ..., nil
}
```

**Files to check and fix where pattern applies:**
- [handlers_health.go](server/internal/api/handlers_health.go): `if err := s.store.Ping(ctx); err != nil` → `if s.store.Ping(ctx) != nil`
- [handlers_auth.go](server/internal/api/handlers_auth.go): Check each `if err :=` where `err` isn't used in body
- [handlers_groups.go](server/internal/api/handlers_groups.go): Same pattern
- [handlers_devices.go](server/internal/api/handlers_devices.go): Same pattern
- [handlers_users.go](server/internal/api/handlers_users.go): Same pattern
- [handlers_sessions.go](server/internal/api/handlers_sessions.go): Same pattern

**Note:** Many S8193 line references point beyond current file lengths (e.g., handlers_devices.go:61 but file has 40 lines). These may be stale alerts that auto-resolve on next scan. Fix the ones that still apply; the rest will clear on rescan.
