# Architecture Decision Records

This page contains the full text of all Architecture Decision Records (ADRs) for the OpenGate project. For a compact index, see [`.claude/decisions.md`](https://github.com/volchanskyi/opengate/blob/dev/.claude/decisions.md) in the repository.

## ADR-001: Wire Protocol — MessagePack + Internally Tagged Enums

**Status**: Accepted | **Phase**: 1

**Decision**: MessagePack encoding with `#[serde(tag = "type")]` internally tagged enums in Rust. Go uses `vmihailenco/msgpack` with matching struct tags.

**Frame format**: `[1-byte type][4-byte BE length][payload]`; Ping/Pong are single-byte (no payload). Handshake uses raw binary encoding (not msgpack), type bytes `0x10–0x15`.

**Why**: JSON is too verbose; protobuf requires schema compilation; custom binary is maintenance-heavy. MessagePack is compact, schema-free, and has good Rust+Go library support.

**Consequences**: Cross-language compat verified in CI via `make golden` (golden files in `/testdata/golden/`, generated by Rust, verified by Go).

## ADR-002: Golden File Tests for Cross-Language Compatibility

**Status**: Accepted | **Phase**: 1

**Decision**: Rust generates canonical binary fixtures ("golden files") in `/testdata/golden/`. Go integration tests load and deserialize them to verify byte-for-byte compatibility.

**Why**: Two independent implementations of the same protocol will silently diverge without a shared contract test.

**Consequences**: Run `make golden` after any protocol change. Go→Rust direction not yet covered (see `techdebt.md`). Golden files are committed.

## ADR-003: SQLite + WAL Mode via Pure Go Driver

**Status**: Superseded by [ADR-014](adr/ADR-014-postgres-migration.md) | **Phase**: 2

**Decision**: `modernc.org/sqlite` (pure Go, no CGo), WAL mode, `foreign_keys` pragma, `MaxOpenConns(1)`.

**Why**: No external DB dependency; pure Go avoids CGo issues for aarch64-musl cross-compilation (OCI ARM64 deployment).

**Consequences**: Single writer at a time (see `techdebt.md`). Phase 13 will replace with PostgreSQL.

## ADR-004: ECDSA P-256 mTLS with Self-Signed CA + CSR Enrollment

**Status**: Accepted | **Phase**: 2

**Decision**: Server is its own CA (10yr P-256 cert). Agents enroll via PKCS#10 CSR posted to `/api/v1/enroll/{token}`; server signs and returns `cert_pem + ca_pem`. TLS 1.3 minimum. `devices.group_id` is NULLABLE with ON DELETE SET NULL — agents register without a group.

**Why**: Mutual authentication without a public PKI dependency. Agents run on untrusted networks.

**Consequences**: CA key loss = full re-enrollment. One-time tokens required for enrollment (issued by admin).

## ADR-005: QUIC mTLS via quic-go (Server Opens Control Stream — Workaround)

**Status**: Accepted (workaround in place) | **Phase**: 4

**Decision**: `quic-go v0.59` for agent transport. Due to a quic-go bug where `AcceptStream` doesn't work with mTLS client certs, the **server opens the control stream** instead of the agent.

**Why**: QUIC gives multiplexed, low-latency transport with no head-of-line blocking.

**Consequences**: Workaround holds up to ~20k agents. Must revert when quic-go fixes mTLS `AcceptStream`. See `techdebt.md` and `plans/quic-stream-ownership-fix.md`.

## ADR-006: Platform Traits with Null Implementations for Headless Environments

**Status**: Accepted | **Phase**: 5

**Decision**: Agent capabilities are Rust traits (`ScreenCapture` with `async_trait`, `InputInjector` and `ServiceLifecycle` as sync/object-safe). Null implementations used in CI and headless containers.

**Why**: Agent must not panic on missing display/input hardware.

**Consequences**: `async_trait` adds a small per-call allocation for screen capture (acceptable at ~30fps). Windows platform is currently all stubs (see `techdebt.md`).

## ADR-007: VAPID Web Push with Keys Persisted in dataDir

**Status**: Accepted | **Phase**: 10

**Decision**: Server generates VAPID keypair on first run, persists to `{dataDir}/vapid.json`. `Notifier` interface decouples push logic from HTTP handlers.

**Why**: Standard Web Push without a third-party push service dependency.

**Consequences**: Changing VAPID keys invalidates all existing browser subscriptions. `dataDir` must be a Docker volume mount.

## ADR-008: aarch64-unknown-linux-musl Cross-Compilation via `cross`

**Status**: Accepted | **Phase**: CD-C

**Decision**: `cross` tool (Docker-based) for Rust → `aarch64-unknown-linux-musl`. Go uses `GOARCH=arm64` natively. Produces fully static binaries.

**Why**: Production target is Oracle Cloud ARM64 Always Free. CI runners are x86_64.

**Consequences**: Docker daemon required in CI. SELinux systems need `restorecon` after OTA binary replacement (fixed 2026-03-26 in `update.rs`).

## ADR-009: Cosign Keyless Signing for Container Images

**Status**: Accepted | **Phase**: CD-E

**Decision**: Keyless signing via GitHub Actions OIDC (Sigstore/cosign). Deploy scripts verify cosign signature before stopping containers. SBOM attestation attached to images.

**Why**: Supply chain security without managing signing keys. GitHub OIDC identity ties signatures to the CI pipeline, eliminating key rotation burden.

**Consequences**: Requires cosign installed on VPS (auto-installed by deploy script). Verification adds ~5s to deploy. Signature verification failure blocks deployment.

## ADR-010: Separate Device Hardware Table + Hardware Control Messages

**Status**: Accepted | **Phase**: —

**Decision**: On-demand hardware inventory stored in a dedicated `device_hardware` table (1:1 with `devices`, CASCADE delete). Three new control messages: `RestartAgent` (server → agent, triggers exit code 42 for systemd auto-restart), `RequestHardwareReport` (server → agent), and `HardwareReport` (agent → server, contains CPU model/cores, RAM, disk total/free, network interfaces as JSON).

**Why**: Hardware data is large, infrequently updated, and orthogonal to the core `devices` row. Storing it in a separate table avoids bloating every device query. On-demand collection (vs. periodic) reduces bandwidth for fleets with hundreds of agents. Agent restart via control message + exit code 42 leverages systemd `RestartForceExitStatus` for zero-downtime restarts without SSH access.

**Consequences**: Two REST endpoints added: `POST /api/v1/devices/{id}/restart` and `GET /api/v1/devices/{id}/hardware` (returns 200 with cached data or 202 when requesting from an offline/in-progress agent). Migration 009 creates the table. The `HardwareReport` payload includes a `network_interfaces` JSON array with name, MAC, IPv4, and IPv6 per interface.

## ADR 011 — On-Demand Device Logs via Control Path

**Status:** Accepted  
**Date:** 2026-04-02

### Context

Operators need visibility into agent-side logs for troubleshooting without SSH access to the device. Logs are written by `tracing-subscriber` into daily-rotated files under `/var/log/mesh-agent/`. We need a mechanism to retrieve, filter, and display these logs through the web UI.

### Decision

Retrieve device logs on demand via the existing QUIC control path using three new control messages:

1. **`RequestDeviceLogs`** (Server -> Agent) — carries filter parameters: `log_level`, `time_from`, `time_to`, `search`, `log_offset`, `log_limit`.
2. **`DeviceLogsResponse`** (Agent -> Server) — returns `log_entries` (array of `LogEntry`), `total_count`, and `has_more` for pagination.
3. **`DeviceLogsError`** (Agent -> Server) — returns an `error` string if the agent cannot read logs.

Each `LogEntry` contains: `timestamp`, `level`, `target`, `message`.

**Server-side caching:** Retrieved log entries are stored as individual rows in a `device_logs` table with a **5-minute cache TTL**. The `HasRecentLogs` store method checks whether cached data is fresh enough to serve directly, avoiding repeated round-trips to the agent.

**REST API:** `GET /api/v1/devices/{id}/logs` with query parameters for filtering (level, from, to, search) and pagination (offset, limit). Returns:
- **200** — cached data served from the database
- **202** — request forwarded to the agent, client should poll again
- **404** — device not found or device is offline with no cached data

**Agent-side `LogCollector`:** Reads daily-rotated log files from `/var/log/mesh-agent/`, parses the `tracing-subscriber` format, applies level/time/keyword filters, and supports offset/limit pagination.

**Frontend:** `DeviceLogs` React component with level dropdown, search input, fetch button, color-coded log levels, and "Load More" pagination.

### Consequences

- No streaming — logs are fetched on demand and cached. Suitable for troubleshooting, not real-time monitoring.
- 5-minute TTL is a trade-off between freshness and control-path load. Can be tuned later.
- SQL-level filtering on cached rows enables fast re-queries within the TTL window without contacting the agent.
- Individual rows (not a single JSON blob) allow efficient filtering by level and timestamp at the database layer.
- Golden file tests cover the three new control messages to ensure Rust/Go encoding compatibility.

## ADR-012: SonarCloud Quality Gate as Hard Merge Block

**Status**: Accepted | **Phase**: —

**Decision**: Run SonarCloud analysis as a required CI job gating `merge-to-main`. The `sonarcloud` job aggregates coverage artifacts from all three language test jobs (Go, Rust, Web) and runs `SonarSource/sonarqube-scan-action@v7` against the full codebase with `-Dsonar.qualitygate.wait=true`, so the job polls SonarCloud until the gate resolves and fails the step if any condition is breached. Findings live in the SonarCloud.io console — there is no SARIF export into GitHub Code Scanning (a SARIF upload step existed briefly and was removed in commit 9236826 because dismissed-fingerprint matching kept new alerts invisible). Quality gate thresholds are configured in the SonarCloud UI on a Clean-as-You-Code model (all conditions apply to *new code* only):

- Coverage on new code ≥ 80%
- Reliability / Security / Maintainability rating A on new code (implicitly forbids new bugs, vulnerabilities, and code smells — any such issue flips the rating)
- 100% of new security hotspots reviewed
- Duplicated lines on new code < 3%

Overall project coverage is **not** part of the SonarCloud gate. Instead it is enforced at the per-language level: `Go Unit Tests`, `Rust Tests`, and `Web Unit Tests` each fail at < 80% line coverage (Go via `go tool cover` bash check, Rust via `cargo llvm-cov --fail-under-lines 80`, Web via `coverage-summary.json` bash check). `/precommit` reproduces the same checks before push. So three independent layers together guarantee ≥ 80% overall **and** ≥ 80% on new code, even though the SonarCloud gate only enforces the latter.

**Why**: Per-language thresholds alone let regressions slip through code-quality dimensions SonarCloud tracks (duplication, cognitive complexity, security hotspots) that `go test -cover` / `llvm-cov` / `vitest --coverage` don't measure. Running SonarCloud as a hard gate (not informational) forces fixes on `dev` rather than letting them accumulate on `main`.

**Consequences**: Adds ~1–2 minutes to CI wall-clock time. SonarCloud outages block merges — mitigated by `workflow_dispatch` bypass and the fact that scheduled runs skip the scan. The `sonar-project.properties` file at repo root is the single source of truth for project key and exclusion paths.
